Write Better Java Unit Tests Using These Less Known Libraries

31 Aug 2019 in Blog

Use Hamcrest, Awaitility and Pitest for your java unit tests

If you try to Google for the best Java unit testing frameworks and libraries, you will get a bunch of famous names such as JUnit, Mockito, TestNG, Cucumber, Spring Test, etc. However, there are 3 libraries that equally valuable and yet they receive less attention.

Hamcrest

To put it simple, Hamcrest allows you to write assertions that read like the English language. It makes the test code declarative. It makes assertions like the below possible

assertThat(testString, is(equalTo("expected string")));
assertThat(testString, is(not(equalTo("un-expected string"))));
assertThat(person, isA(Person.class));
assertThat(personsList, hasItem(personX));

The way Hamcrest achieves this is by always using the assertThat with a set of matchers that the library provides. A matcher is a predicate. It returns a true or a false based on the input parameters. Some of the most useful matchers are:

//general
is(), equalTo(), not(), nullValue(), isA(), anyOf(), allOf(), 

//collections
hasItem(), hasSize()

//String
startsWith(), startsWithIgnoringCase(), containsString()

// Numbers
greaterThanOrEqualTo(), lessThan()

You can also write your own matchers to perform custom assertions.

Hamcrest integrates with all versions of JUnit and with TestNG so you can start using the library right away after adding the test dependency in your product

<!-- maven -->
<dependency>
    <groupId>org.hamcrest</groupId>
    <artifactId>java-hamcrest</artifactId>
    <version>2.0.0.0</version>
    <scope>test</scope>
</dependency>

// gradle
testCompile group: 'org.hamcrest', name: 'java-hamcrest', version: '2.0.0.0'

Get started with Hamcrest here.

Awaitility

Writing high quality unit tests is hard. Writing high quality unit tests for asynchronous code is much harder. Awaitility promises to make it easier for us to write clean and readable tests for async code. In its simplest form, you can use Awaitility as follows

await().until(someConditionIsMet()); // someConditionIsMet returns a Callable<Boolean>

If the condition is not met after a certain time (default 10 seconds) this call will throw an exception and the test will fail. Otherwise, the test passes asserting the condition is met.

Suppose that we are adding a person to a list through an async call. We can assert that the person is added as below

addToPersonsList(new Person()); // this method calls an asynchronous system

await().until(() -> personsList.size() == 1);

You can change the timeout we wait for results using the methods atLeast() and atMost() that the library provides making the below a valid

await().atLeast(1, SECONDS).and().atMost(2, SECONDS).until(() -> personsList.size() == 1);

Awaitility uses polling to verify that the condition is met. The polling period and its algorithm can be configured. The library also provides other advance usages that you can explore in their Usage page.

Awaitility integrates with JUnit and can be used by adding the below dependencies

<!-- maven -->
<dependency>
      <groupId>org.awaitility</groupId>
      <artifactId>awaitility</artifactId>
      <version>4.0.0</version>
      <scope>test</scope>
</dependency>

// gradle
testCompile 'org.awaitility:awaitility:4.0.0'

Pitest

Pitest is a mutation testing tool for the java language. It is different from the other libraries because Pitesst is not used in writing tests, but in making sure that the tests we write will catch bugs if the production code ever changes.

The way pitest works is as follows:

  1. It creates copies of the bytecode of our program
  2. It changes the copies in small but different ways
  3. It runs the unit test suite on each of these copies.
  4. It asserts that the unit tests will fail on each of the manipulated copies. If the test did not fail it means that there is a change that could happen to the production code that will not be caught by the unit tests

So Pitest is a library that will allow us to assert the quality of our test by making sure that our tests fail if a change happens to the production code. Other test quality metrics that are traditionally used such as code coverage are still important to measure but are not enough to ensure we write high quality, valid tests.

To use Pitest, you need to add the below dependency and then use the plugin to generate the Pitest reports as part of the build lifecycle.

<!-- maven -->
<dependency>
    <groupId>org.pitest</groupId>
    <artifactId>pitest-parent</artifactId>
    <version>1.4.9</version>
    <type>pom</type>
</dependency>

// gradle
compile group: 'org.pitest', name: 'pitest-parent', version: '1.4.9', ext: 'pom'

Summary

  • Hamcrest
    • Write declarative assertions making the tests cleaner and more readable
  • Awaitility
    • Test Asynchronous systems by making tests wait for the results of the async calls in an intuitive way.
  • Pitest
    • Make sure that your tests will catch bugs in the code by changing the production code and asserting tests will fail with those changes
